# Yantra Paper Generation Pipeline: User Guide

This document provides an in-depth, user-centric guide on how to leverage the Amsha Agent System to automatically generate publication-ready, Scopus-indexed (ESWA-compliant) research papers directly from your codebase.

The paper generation process is managed by a triad of orchestrator agents, meant to be executed sequentially:
1. **Dakini:** The Configuration Scaffolder
2. **Lutapi (Journal Master):** The Deep Codebase Analyst
3. **Kutusan:** The Journal Section Drafter

---

## 1. Dakini (The Configuration Builder)

**Purpose:** Before any paper can be written, the agents need to know what code exists. Dakini scans your repository, detects your python packages/modules, and automatically generates a YAML configuration file (`.agent/paper_config.yaml`) that maps out priorities and inclusion flags.

**When to use:** Use Dakini at the very beginning of the paper writing process, or whenever you add a completely new structural module to your codebase that you want included in the paper.

### Dakini Trigger Commands

| Command Trigger | User Intent | Action / Output |
| :--- | :--- | :--- |
| `"Dakini, generate paper config"` | Standard generation | Generates `.agent/paper_config.yaml` scanning the local `src/` modules. |
| `"Dakini, generate library paper config"` | Multi-library generation | Generates config targeting specific external libraries (e.g., Yantra, Amsha). |

---

## 2. Lutapi / Journal Master (The Deep Code Analyst)

**Purpose:** Lutapi does the heavy lifting of extracting technical intelligence *before* the paper is written. It reads the code and generates dense markdown artifacts summarizing the Architecture, Algorithms (LaTeX), Performance metrics, Research Gaps, and Academic Novelty. 

**Workflow:** Lutapi must be run **Phase-Wise**. Attempting to analyze the entire codebase in one prompt will result in context exhaustion and failure. You must analyze modules one at a time, followed by cross-module analysis.

### Lutapi Trigger Commands

| Trigger Command | Execution Phase | What it Generates / Output |
| :--- | :--- | :--- |
| `"Lutapi, analyze module [module_name] for the paper"` | **Phase 1: Module Analysis**<br>*(Run once per module)* | 5 files in `docs/paper/Yantra/[module_name]/`: <br>`mathematics.md`, `architecture.md`, `novelty.md`, `gaps.md`, `summary.md`. |
| `"Lutapi, run cross-module analysis for the paper"` | **Phase 2: Internal Synergy**<br>*(Run after all modules)* | Analyzes how local modules interact. Outputs 8 files in `cross_module/`: `interactions.md`, `dependencies.md`, `patterns.md`, `novelty.md`, `architecture.md`, `mathematics.md`, `gaps.md`, `consistency_check.md`. |
| `"Lutapi, run cross-library analysis for the paper"` | **Phase 3: External Synergy**<br>*(Run if using external libs)* | Analyzes how Yantra interacts with Amsha/Bodha. Outputs to `cross_library/`. |
| `"Lutapi, generate user input templates for the paper"` | **Phase 5: User Input Templates**<br>*(Run after cross-module)* | Auto-generates `app_execution_results.md` (guided template) and `practical_implications.md` (auto-filled, needs user verification) in `user_inputs/`. |
| `"Lutapi, show paper generation progress"` | **Tracking** | Reads `.progress.yaml` to show you which modules have been successfully analyzed so far. |

> **Crucial Note on Lutapi:** Lutapi does **not** write the final paper prose. It generates the *raw facts* (the ground truth evidence) that Kutusan will later use to draft the prose without hallucinating.

---

## 2.5 The Human-in-the-Loop: User Input Templates

Lutapi's Phase 5 auto-generates two crucial files in `docs/paper/Yantra/user_inputs/` that you **must** interact with before Kutusan can build Sections 6 and 7 of your paper.

### 1. `app_execution_results.md` (Guided Template)
- **Used for:** Section 6 (Results and Discussion)
- **What it is:** A smart skeleton pre-filled with suggested metrics (e.g., F1, Latency) and baselines inferred from your codebase's ML analysis.
- **Your Job:** You must physically run your application, collect the real metrics, and **fill in all the `[USER]` placeholders** with actual empirical numbers. Kutusan cannot hallucinate empirical performance data!

### 2. `practical_implications.md` (Auto-Generated Details)
- **Used for:** Section 7 (Practical and Managerial Implications)
- **What it is:** Auto-generated text containing specific claims about industry applicability, cost-efficiency, and deployment inferred directly from your code's architecture and configuration.
- **Your Job:** You must open this file, **review each claim, and verify it** by checking the `[ ]` boxes to `✅` (or rejecting/editing claims that don't fit). Kutusan will only use the claims you explicitly verify.

---

## 3. Kutusan (The Section Orchestrator)

**Purpose:** Kutusan takes the raw facts generated by Lutapi and orchestrates a 5-stage pipeline to formally compose the final `FINAL_JOURNAL_REPORT.md` section by section. It ensures the writing adheres to the strict 11-section format required by the *Expert Systems with Applications* (ESWA) journal.

**Workflow:** Kutusan runs a rigorous 5-stage internal pipeline for every section:
1. `paper-planner`: Writes the strict operational rules (e.g., "Must be 200 words, no citations").
2. `paper-analyzer`: Pulls the exact facts needed from Lutapi's output.
3. `external-researcher` / `deep-analyzer`: Enriches data with Q1 Scopus citations or Big-O complexity tables. *(Runs conditionally)*.
4. `paper-drafter`: Converts facts into Scopus-level academic prose.
5. `paper-reviewer`: Fact-checks the drafted prose against the analysis to guarantee zero hallucination and proper tone.

### Kutusan Trigger Commands

| Trigger Command | Intent | Internal Agents Triggered | Output Target |
| :--- | :--- | :--- | :--- |
| `"Kutusan, initialize the final journal report outline"` | Initial Setup | None | Scaffolds the empty 11-section `FINAL_JOURNAL_REPORT.md`. |
| `"Kutusan, generate the [Section Name] for the paper"` | **Drafting a Section**<br>*(e.g., "Introduction", "Abstract")* | `planner` → `analyzer` → `augmenter` (optional) → `drafter` → `reviewer` | Five artifacts ending in `verified_[Section].md` saved inside `docs/paper/Yantra/drafts/details/[N]. [Section Name]/`. |
| `"Kutusan, finalize the journal paper"` | Final Assembly | `eswa-compliance-checker` | Runs a global constraint check (word limits, 35-45 refs, 60-70% citation recency). If passed, concatenates all verified sections into the final PDF/Markdown. |

### The 11 ESWA Sections

When calling `"Kutusan, generate the [Section Name]"`, you must proceed through the following 11 strictly mandated ESWA sections sequentially:

1. Abstract
2. Introduction
3. Related Work
4. Problem Definition
5. Proposed Methodology
6. Results and Discussion
7. Practical and Managerial Implications
8. Limitations and Future Work
9. Conclusion
10. Acknowledgment
11. Reference

## 🔄 Summary Complete Workflow Example

If you were starting from scratch, your chat sequence with the AI over a few days would look exactly like this:

1. **You:** `"Dakini, generate paper config"`
2. **You:** `"Lutapi, analyze module orchestration for the paper"`
3. **You:** `"Lutapi, analyze module monitoring for the paper"`
4. **You:** `"Lutapi, run cross-module analysis"`
5. **You:** `"Lutapi, generate user input templates for the paper"`
6. **You:** *(Review & fill `app_execution_results.md`, verify `practical_implications.md`)*
7. **You:** `"Kutusan, initialize the final journal report outline"`
8. **You:** `"Kutusan, generate the Abstract for the paper"`
9. **You:** `"Kutusan, generate the Introduction for the paper"`
10. *(Continue section by section...)*
11. **You:** `"Kutusan, finalize the journal paper"`
